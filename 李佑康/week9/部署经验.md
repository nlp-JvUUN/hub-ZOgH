# vLLM-Ascend 部署经验总结

我们有过实际部署 vLLM-Ascend 的经验。相比直接使用 Transformers 进行推理，vLLM-Ascend 在推理速度上有明显提升，但它并不是完全开箱即用的方案。即使按照官方文档部署，过程中仍然可能遇到环境、算子、参数组合和性能调优等问题，需要结合模型特点和硬件条件反复验证。

## 1. 总体经验

vLLM-Ascend 的部署重点不只是“能跑起来”，更重要的是找到适合当前模型和硬件的参数组合。官方文档中提供的配置通常只能作为起点，并不一定能达到最优效果。

实际调优时需要重点关注以下指标：

- TTFT：首 token 延迟，主要受 prefill 阶段影响。
- TPOT：每个输出 token 的平均耗时，主要影响持续生成阶段的吞吐和响应速度。
- KV cache 命中率：会影响长上下文、多轮对话和高并发场景下的性能。
- 算力利用率：需要观察 prefill 和 decode 两个阶段是否存在资源空转。

## 2. PD 分离与 PD 混补的选择

PD 分离并不是所有场景下的最优解，也不能简单理解为 PD 混补的上位替代。选择哪种方式，需要结合模型规模、prefill/decode 开销比例以及跨节点数据传输成本来判断。

根据我们的经验，120B 左右的模型更推荐使用 PD 混补。这样可以减少 prefill 和 decode 阶段之间的数据频繁传输，从而降低 TPOT，整体表现通常更稳定。

对于 GLM-5.1-w4a8 这类模型，实际测试中发现它们在 prefill 阶段会消耗更多算力和时延，因此更适合采用多 P 对 1 D 的部署方式。例如使用 2P1D 或 3P1D，可以更充分地利用算力，缓解 prefill 阶段的瓶颈。

而对于 Minimax 2.7 这类模型，更建议采用 1P1D 的方式。它的 prefill 和 decode 负载相对更均衡，过多拆分 P 节点不一定能带来收益，反而可能增加调度和通信成本。

## 3. Mooncake 与 KV Cache Offload

引入 Mooncake 后，可以将 KV cache offload 到 DDR 或磁盘中，从而提高 KV 命中率，降低 TPOT。在长上下文或多轮对话场景中，这类优化比较有价值。

但需要注意，offload 并不一定总是正收益。如果磁盘读写速度较慢，KV cache 的读取延迟可能会抵消命中率提升带来的收益，甚至导致整体性能下降。因此在开启 Mooncake 前，需要重点评估磁盘 I/O 能力，并通过压测观察端到端延迟变化。

## 4. MTP 参数建议

对于量化后的模型，不建议将 MTP 设置得过高。实际经验中，量化模型对较高 MTP 的接受度有限，参数过高可能无法带来稳定收益，还可能引入额外的性能波动。

更稳妥的方式是从较小的 MTP 值开始测试，逐步观察吞吐、延迟和输出稳定性，再决定是否继续提高。

## 5. 算子与并行配置调优

FlashComm、MC2 等算子是否开启，以及 SP、CP 等并行策略如何配置，都需要根据具体模型和硬件环境进行测试。它们没有固定的最优组合。

调优时建议按以下顺序进行：

1. 先确定基础部署方式，例如 PD 分离、PD 混补、1P1D 或多 P1D。
2. 再调整并行策略，例如 TP、SP、CP 等配置。
3. 最后测试算子开关，例如 FlashComm、MC2 等。
4. 每次只改动一类参数，并记录 TTFT、TPOT、吞吐和显存占用变化。

## 6. 总结

vLLM-Ascend 能显著提升推理性能，但实际部署中有大量细节需要调试。不同模型之间的最优配置差异很大，不能直接复用一套参数。

我们的经验是：先根据模型的 prefill 和 decode 特征选择部署架构，再围绕 KV cache、MTP、算子开关和并行策略逐步压测。只有结合真实业务负载和硬件环境，才能找到稳定且性能较优的配置。
