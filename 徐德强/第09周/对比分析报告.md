## 二、约束解码 Demo 结果

所有 Demo 均通过 OpenAI 兼容客户端调用 `http://localhost:8000/v1`，模型名 `qwen2.5-0.5b`。

### 2.1 `demo_guided_choice.py` — 枚举约束

**场景**：金融意图路由（查股价/查财报/查新闻/对比分析/其他）共 12 条测试

| 指标 | 裸 prompt | guided_choice |
|------|:---------:|:-------------:|
| 输出合法率 | 12/12 (100%) | 12/12 (100%) |
| 预测正确率 | 5/12 **(42%)** | 5/12 **(42%)** |
| 平均延迟 | **0.598s** | **0.959s** (+60%) |

**结论**：
- **预测正确率（42%）是模型理解的瓶颈，不是输出格式的瓶颈**——guided_choice 保证输出不跑出枚举范围，但模型本身分不清"查财报"和"对比分析"的区别
- **延迟增加了 60%**（0.598s → 0.959s），是因为 FSM 在解码每步都要检查合法 token 集合，0.5B 模型的计算时间短，约束解码的额外开销占比就大
- **选型建议**：如果模型本身就能稳定输出合法枚举（如这里 100% 合法），加 guided_choice 的意义不大；但模型不够稳定时（小模型或高 temperature 场景），它能兜底保证下游解析器不崩

---

### 2.2 `demo_guided_regex.py` — 正则约束

**场景**：日期标准化（YYYY-MM-DD）+ A 股代码抽取（6 位数字）

| 场景 | 指标 | 裸 prompt | guided_regex |
|:----|------|:---------:|:------------:|
| 日期标准化 | 格式合法率 | **3/6 (50%)** | **6/6 (100%)** |
| 股票代码 | 格式合法率 | 5/5 (100%) | 5/5 (100%) |

**典型案例**：
- `"三月三号我去北京"` → 裸 prompt 输出 `03-03` ❌，guided_regex 输出 `0331-00-00` ✅（格式合但语义错）
- `"六零零五一九"` → 裸 prompt 和 guided_regex 都输出 `600595`（格式对但代码错）

**结论**：
- **日期场景差距显著（50% → 100%）**——"三月三号"、"2023/12/1 下午开会"这类自然语言日期，裸 prompt 会混入时间或格式错乱，guided_regex 强制每一步只生成 `\d{4}-\d{2}-\d{2}` 格式的字符，彻底根治
- **股票代码场景两边都好**（100%）——因为股票代码本身就是 6 位数字，裸 prompt 也很容易遵守
- **关键区分：格式合法 vs 语义正确**——`"0331-00-00"` 格式完全匹配正则，但日期是错的；`"600595"` 是合法 6 位代码但不是用户要的 `"600519"`。正则约束不解决语义问题

### 2.3 `demo_guided_json.py` — JSON Schema 约束

**场景**：财报意图抽取（公司/年度/指标三元组），9 条测试，三路对比

| 指标 | 裸 prompt | response_format | guided_json |
|------|:---------:|:---------------:|:-----------:|
| 合法 JSON | 9/9 (100%) | 9/9 (100%) | 9/9 (100%) |
| 字段齐全 | 9/9 (100%) | 9/9 (100%) | 9/9 (100%) |
| year 在 2015~2025 | 9/9 (100%) | 9/9 (100%) | 9/9 (100%) |
| **metric 在枚举内** | **8/9 (89%)** | **8/9 (89%)** | **9/9 (100%)** |
| **jsonschema 完全通过** | **8/9 (89%)** | **8/9 (89%)** | **9/9 (100%)** |

**典型案例**：`"ICBC 2023 营收"` → 裸 prompt 输出 `"metric": "营业收入"`（全称不在枚举中），guided_json 强制为 `"metric": "营收"`。

**结论**：
- **任务简单时裸 prompt 和 response_format 差距不大**（89% vs 89%）——因为财报三元组（公司/年度/指标）的结构化程度足够清晰，模型本身就能稳定输出合法 JSON，response_format 的"保证是 JSON"在这里没有额外收益
- **差距集中在枚举约束**——唯一的失败案例是 `"ICBC 2023 营收"` → 模型输出 `"营业收入"`（财务全称），不在枚举 `["营收", "净利润", ...]` 中。guided_json 通过 FSM 在解码阶段屏蔽了"营业"这两个 token，强制模型只能从枚举里选
- **三层指标的价值**：分层评估（合法 JSON → 字段齐全 → schema 完全通过）让我们能精确定位失败发生在哪一层——这里走到了"字段齐全"但死在"枚举值匹配"

### 2.4 `demo_response_format.py` — OpenAI 标准方式

**场景**：新闻情感分类（positive/negative/neutral）+ 置信度 + 关键词，5 条测试

| 指标 | 裸 prompt | response_format |
|------|:---------:|:---------------:|
| 合法 JSON | 5/5 (100%) | 5/5 (100%) |
| 全部字段齐全 | 5/5 (100%) | 5/5 (100%) |

**结论**：
- **所有指标都是 100%**——意味着新闻情感分类这个任务对 Qwen2.5-0.5B 来说太简单了，裸 prompt 已经足够好
- **response_format 没有发挥优势**，因为任务本身不涉及复杂 schema、嵌套对象、枚举约束等容易出错的地方
- **这个结果印证了一个工程原则**：约束解码不是"用了就更好"，而是"用在该用的地方才有价值"。简单场景加约束只是增加代码复杂度和额外延迟，收益为零

### 2.5 `demo_function_call.py` ★ 核心演示

**场景**：两个工具（`get_stock_quote` + `create_order`），各 50 条测试，三路对比

#### 工具 1：get_stock_quote（股价查询）

| 指标 | 裸 prompt | response_format | guided_json |
|------|:---------:|:---------------:|:-----------:|
| JSON 语法合法 | 50/50 (100%) | 50/50 (100%) | 50/50 (100%) |
| 必选字段齐全 | 50/50 (100%) | 50/50 (100%) | 50/50 (100%) |
| **完整 schema 通过 ★** | **46/50 (92%)** | **46/50 (92%)** | **50/50 (100%)** |
| 平均延迟 | 1.635s | 1.784s | 4.189s |

#### 工具 2：create_order（电商下单）

| 指标 | 裸 prompt | response_format | guided_json |
|------|:---------:|:---------------:|:-----------:|
| JSON 语法合法 | 50/50 (100%) | 50/50 (100%) | 50/50 (100%) |
| 必选字段齐全 | 50/50 (100%) | 50/50 (100%) | 50/50 (100%) |
| **完整 schema 通过 ★** | **27/50 (54%)** | **27/50 (54%)** | **46/50 (92%)** |
| 平均延迟 | 1.870s | 2.102s | 5.500s |

#### 失败模式总结

| 失败类型 | 发生场景 | 示例 |
|---------|---------|------|
| 正则违规 | `get_stock_quote` | `"symbol": "02699"`（5 位，期待 6 位） |
| 额外字段注入 | `get_stock_quote` | `"isBuy": true`（多出未定义字段） |
| 枚举混淆 | `create_order` | `"priority": "wechat"`（支付方式填到优先级） |
| Integer 范围（guided_json 也失败） | `create_order` | `"quantity": 200`（超过 max=100） |

#### 约束解码工程价值总结

| 工具 | 裸 prompt | response_format | guided_json | 提升 |
|:----|:---------:|:---------------:|:-----------:|:----:|
| get_stock_quote | 92% | 92% | **100%** | +8% |
| create_order | 54% | 54% | **92%** | **+38%** |

**结论**：
- **场景复杂度决定约束解码的价值**：get_stock_quote（92%→100%，+8%）的提升远小于 create_order（54%→92%，+38%），因为 stock 的 schema 只有 4 个字段且约束简单，而 order 包含了正则、枚举、integer 范围等多种约束组合，模型裸跑容易踩坑的地方更多
- **response_format 和裸 prompt 结果完全一致**（92% 和 54%）——因为 response_format 只管"输出是合法 JSON"，不管字段值是否合规。schema 校验失败不是因为 JSON 语法错，而是字段语义错（手机号少一位、枚举值填错、数量超范围），这些 response_format 管不了
- **延迟代价约 2~3×**（stock: 1.6s→4.2s, order: 1.9s→5.5s），比前几个 demo 的 60% 增幅更大——因为 function_call 的 JSON Schema 更复杂，FSM 构建和每步 token 屏蔽的开销也更大
- **guided_json 的边界**：integer 的 `minimum`/`maximum` 范围约束不是 100% 硬屏蔽（order 中 4 条失败），这是 vLLM 0.9.2 的实现局限。生产环境不能完全依赖约束解码，需要额外加一层校验

## 三、吞吐基准测试结果

### 测试设计说明

**为什么选 50 条 prompts？**
- 数量足够产生统计意义，又不至于跑太久（最慢的串行模式需 3 分钟）
- 50 条覆盖了从短到长的各种 prompt 长度，模拟真实业务的混合负载

**为什么长短混合？**
- 短 prompt（"什么是股票？"）→ prefill 快，generate 占主导
- 长 prompt（"详细介绍一下巴菲特的投资理念..."）→ prefill 占比高，对 batch 效率影响大
- 长短混合对 continuous batching 最有利——短请求先完成、释放 slot、新请求补入

**为什么 max_new_tokens=100？**
- 生产环境中多数对话/抽取任务的目标输出在 50~200 token 之间
- 100 token 是常见中间值，既能体现 generate 阶段的吞吐差异，又不会让测试时间过长

**测试流程**：
```
同一个模型（Qwen2.5-0.5B-Instruct）、同一组 50 条 prompts，依次跑 4 种模式：
  [A] Transformers 串行     → for 循环逐条推理
  [B] Transformers batch=8  → 手动 padding + 每 8 条一组推理
  [D] vLLM Server HTTP 并发 → 启动 Server，用 openai 客户端 + 多线程并发
  [C] vLLM 离线批处理       → 用 vLLM 的 LLM.generate() 直接推理
注意：[A][B][C] 需要停掉 Server 释放显存，[D] 需要保持 Server 运行

**生图**：`src/plot_results.py` 加载 outputs/ 下的 JSON 数据独立出图，
不需重跑 50 次实验。用法：`python plot_results.py -c 20`。
```

### 3.1 四路吞吐全对比

| 模式 | 总耗时 | QPS | tokens/s | 相对基准 |
|:----|:-----:|:---:|:--------:|:--------:|
| [A] Transformers 串行 | **180.07s** | **0.28** | 28 | **1×** |
| [B] Transformers batch=8 | **32.93s** | **1.52** | 152 | **5.5×** |
| [D] vLLM Server HTTP 并发（c=20） | **21.93s** | **2.28** | 228 | **8.1×** |
| [C] vLLM 离线批处理 | **14.78s** | **3.38** | 307 | **12.2×** |

从 0.28 QPS 到 3.38 QPS，同样的模型、同样的数据，只是换了一套推理引擎，吞吐就拉开了 **12 倍** 的差距。

---

### 3.2 各模式逐路分析

#### [A] Transformers 串行 — 最朴素的基线（0.28 QPS）

**工作机制**：
```
for 循环:
    1. tokenizer(prompt) → input_ids
    2. model.generate(input_ids) → 输出 tokens
    3. 等前一条完全结束才跑下一条
```

**瓶颈**：
- GPU 利用率极低（估算 ~5~10%），绝大部分时间在等待 Python 循环的下一轮
- 每轮都要重新加载权重到 CUDA kernel（无缓存复用）
- **50 条请求 180 秒**，平均每条 3.6 秒——用户层面完全不可接受

**代表场景**：原型验证、单条调试。生产环境**绝对不能用**。

---

#### [B] Transformers batch=8 — 手动组 batch（1.52 QPS）

**工作机制**：
```
for 每 8 条一组:
    1. tokenizer(pad=True) → padding 到等长
    2. model.generate(batch) → 同时生成 8 条
    3. 早完成的请求等最慢的结束才一起返回
```

**提升 5.5× 的原因**：
- GPU 并行计算优势被利用——8 条同时推理，计算密集型 matmul 接近线性加速
- 权重加载开销被分摊到 8 条请求

**手写 batch 的三大浪费**：
1. **Padding 浪费**：短 prompt（<10 token）要 pad 到 batch 内最长（>50 token），pad token 的 attention 计算白做
2. **拖尾浪费**：短请求（5 token 输出）要等长请求（100 token 输出）结束后才能返回
3. **实现成本**：代码复杂度上升，padding_side、attention_mask、pad_token_id 全要手动处理

**和 vLLM 的差距体现**：手动 batch 虽然比串行好，但 padding 浪费决定了它离理论最优还很远。

---

#### [C] vLLM 离线批处理 — 引擎直连最快路径（3.38 QPS）

**工作机制**：
```
from vllm import LLM
llm = LLM(model_path)
llm.generate(prompts, sampling_params)  # 一行搞定
```

**为什么最快**：
- **PagedAttention**：KV cache 按 block（16 token）分配，不连续内存也不 pad，彻底消除 padding 浪费
- **Continuous batching**：不等 batch 内最长请求结束，短请求一完成就插入新请求，GPU 利用率拉满
- 无网络开销：进程内直接调用，省去 HTTP 序列化/反序列化

**适用场景**：离线批处理、对延迟不敏感的批量推理任务。Python 进程内调用，性能最优但无法跨语言调用。

---

#### [D] vLLM Server HTTP 并发 — 生产环境标准路径（2.28 QPS）

**工作机制**：
```
Server 侧: vLLM Engine + FastAPI + OpenAI 兼容 API
Client 侧: openai 客户端 + ThreadPoolExecutor 并发
请求流:  Client → HTTP POST → FastAPI 路由 → vLLM Engine → 生成 → HTTP Response
```

**8.1× 相对基准的加速来自**：
- Continuous batching + PagedAttention（和 [C] 相同的引擎内核）
- 并发 20 时 Engine 内部 batch size 达到 20，GPU 接近满载

**为什么比 [C] 慢 1.48×**：
- 每请求一次 HTTP 序列化/反序列化（JSON ↔ dict）
- FastAPI 路由匹配 + asyncio 事件循环调度
- Gunicorn/Uvicorn worker 之间的 GIL 争用
- 这是"通用性"的代价——OpenAI 兼容 API 换来跨语言调用能力

**适用场景**：生产部署、需要跨语言调用（Java/Go/Node.js 等）、需要 OpenAI 兼容接口。

### 3.3 Server 并发梯度测试

**目的**：找到 GPU 算力的饱和点，确定生产环境的最佳并发配置

| 并发数 | 总耗时 | QPS | tokens/s | HTTP 开销 | GPU 状态 |
|:-----:|:-----:|:---:|:--------:|:---------:|:--------:|
| 5 | 38.67s | **1.29** | 129 | 2.62× | 🟢 欠载，有闲置 |
| 10 | 23.06s | **2.17** | 217 | 1.56× | 🟡 接近满载 |
| 20 | 21.93s | **2.28** | 228 | **1.48×** | 🔴 饱和，排队中 |

#### 从数据看 Server 内部发生了什么

**并发 5 时（GPU 欠载）**：
- Engine 调度器每次只能收到少量请求，continuous batching 组不出大 batch
- 实际 batch size 可能只有 3~5，GPU 计算单元大量空闲
- **瓶颈在客户端**——请求到达率不够，Server "吃不饱"
- HTTP 开销看起来最大（2.62×）是因为空闲时的固定开销被摊到更少的请求上

**并发 10 时（接近满载）**：
- Engine 调度器能持续拿到 10 条待处理请求
- Continuous batching 动态组 batch，实际 batch size 达到 8~10
- QPS 从 1.29 跳到 2.17，几乎翻倍——**这个区间收益最大**
- HTTP 开销降到 1.56×，因为固定开销被更多的请求分摊

**并发 20 时（GPU 饱和）**：
- GPU 计算资源已满，再多的请求只能在队列里排队
- QPS 只涨了 5%（2.17→2.28），收益进入平台期
- 有利的一面：排队缓冲区保证了 Engine 始终"有活干"
- 不利的一面：P99 延迟会显著上升（部分请求等待时间长）

#### 并发选择的工程权衡

**生产环境建议取饱和点的 60~80%**（即并发 6~8，而非 10 或 20）：
- 避免排队效应——并发 20 时 GPU 利用率最高但 P99 延迟变差
- 留出余量应对流量尖峰——瞬时突发 2× 请求时不至于雪崩
- 实测 QPS 1.7~1.9 就够覆盖绝大多数个人和小团队场景

### 3.4 为什么最大加速只有 12×？（GTX 1660 Ti 的局限）

| 限制因素 | 影响 | 说明 |
|---------|:----:|------|
| Turing 架构不支持 FlashAttention-2 | -30~50% | vLLM 自动降级到 XFormers backend |
| V1 Engine 不支持（CC 7.5 < 8.0） | -10~20% | 回退到 V0 Engine |
| 显存仅 6GB | -50% | 最大 batch size 受限，并发上限低 |
| PCIe 3.0×16（非 4.0） | -5~10% | 模型加载和 prefill 阶段带宽瓶颈 |

**换个说法**：如果这台机器换成 RTX 4060（8GB + Ampere + FlashAttention-2），预期 QPS 可达 8~10（约 30~40× 加速比）。当前 12× 是硬件条件下的合理表现。

### 3.5 四路对比的核心启示

**① 从 [A]→[B] 提升 5.5×：最简单的批处理就能释放数倍潜力**
  - 串行推理时 GPU 绝大部分时间在空转等待 CPU 循环
  - 说明：**使用模型前的第一个优化永远是"别串行"**
  - 代价是代码复杂度上升 + padding 浪费

**② 从 [B]→[D] 提升 1.5×：Continuous batching 消除了 padding 浪费**
  - [B] 的手动 batch 有 30~50% 的 token 是 pad token，白算
  - [D] 的 PagedAttention 按需分配，0 padding 浪费
  - 说明：**手动 batch 只是"能跑"，框架级优化才能榨干 GPU**

**③ 从 [D]→[C] 提升 1.48×：HTTP 层是有成本的**
  - [C]=3.38 QPS 是引擎极限，[D]=2.28 QPS 是 Server 吞吐
  - 差值是 HTTP 序列化 + FastAPI 路由 + 并发锁争用的固定开销
  - 说明：**通用性（OpenAI 兼容 API）和性能之间存在不可避免的 trade-off**

**④ 总体看，四条路径代表了四种部署层级**
  - [A] 原型验证 → [B] 少量并发 → [D] 生产部署 → [C] 极致优化
  - 大部分生产场景选 [D]（Server 路径）就够了，没必要追求最后的 1.48×

## 五、核心结论

### 5.1 约束解码

1. **guided_json（guided_choice/guided_regex）** — 100% 保证输出格式合法，工程价值在小模型上尤为显著（create_order：54% → 92%）
2. **约束解码不能提高语义正确率** — 模型能力决定理解，约束只保证格式
3. **约束解码有代价** — 延迟增加 60%~160%（FSM 构建 + token 屏蔽）
4. **`response_format` 介于两者之间** — 保证 JSON 语法，不保证字段语义

### 5.2 吞吐性能

1. **vLLM 比 Transformers 串行快 12.2×**（3.38 vs 0.28 QPS）
2. **Server 路径比离线推理慢约 1.5×**（HTTP 序列化开销）
3. **GTX 1660 Ti 6GB 能流畅跑 0.5B 模型** — 推理完全可用

### 5.3 选型建议

| 场景 | 推荐模式 |
|------|---------|
| 简单分类/路由 | `guided_choice` |
| 严格格式输出（日期/代码/电话） | `guided_regex` |
| 结构化抽取（JSON） | `guided_json` |
| 跨厂商兼容 | `response_format` |
| 高吞吐生产环境 | vLLM Server + 适量并发 |
| 离线批处理 | vLLM 离线 API（最快） |

## 六、遗留问题

- **guided_json integer 范围约束不彻底** — `create_order` 中 quantity=200 仍能通过 guided_json 输出（共 4 条），说明 vLLM 对 `minimum`/`maximum` 约束不是 100% 硬屏蔽
- **GTX 1660 Ti（Turing CC 7.5）** — 不支持 FlashAttention-2，vLLM 自动降级到 XFormers，若换 RTX 30/40 系列吞吐预计更高
- **模型太小** — 0.5B 在复杂理解任务上（意图分类仅 42% 正确率）有天花板，生产环境建议至少 7B+
