## 三、关键发现与分析

### 1. 数据集复杂度决定了模型表现上限

| 对比维度 | cluener (ner_demo) | 人民网 (ner_demo2) |
|------|:---:|:---:|
| 实体类型数 | 10 | **3** |
| BERT+Linear F1 | ~79% | **93.29%** |
| BERT+CRF F1 | ~72.5% | **93.70%** |

人民网 F1 比 cluener 高出 **14-21 个百分点**，核心原因是：
- **类别数减少 70%**（10→3）：标签空间从 21 缩小到 7，分类难度骤降
- **训练数据翻倍**（10.7K→20.9K）：模型学到更丰富的上下文模式
- **实体类别语义边界清晰**：PER/ORG/LOC 三类在中文中语义区分度远高于 cluener 的细粒度类别（如 "government" vs "organization"）

### 2. CRF 相比 Linear 的优势微弱

- F1 提升仅 **+0.41%**（93.29%→93.70%）
- 非法序列：76 vs 86，差异仅 10 条

**分析**：当基础 F1 已高达 93%+ 时，CRF 的边际收益自然递减。此外仅训练 1 epoch，CRF 的转移矩阵尚未充分收敛（训练 loss 4.37 远高于 Linear 的 0.08，CRF 收敛更慢）。**延长训练到 3-5 epochs 后 CRF 的优势会更明显**，且非法序列有望降至接近 0。

> ⚠️ 注意：cluener 上 CRF F1 反而低于 Linear（72.5% vs 79%），说明 CRF 在小数据集/短文本场景下可能过拟合转移约束。而人民网的大数据量+长文本场景对 CRF 更友好。

### 3. 人名识别近乎完美，组织机构是瓶颈

```
PER:  F1 = 97.57%  ← 人名模式固定（姓+名，2-3字），BERT 极易学习
LOC:  F1 = 94.03%  ← 地名有常见后缀（省/市/县/区/山/河）
ORG:  F1 = 88.22%  ← 组织机构名称多样、长度不一，且常含嵌套地名
```

ORG 的 recall 偏低（91.36% vs PER 的 97.85%），说明模型倾向于漏召回长尾机构名。CRF 略微改善了这一情况（ORG recall 89.84% → 略降，但 precision 87.79% → 提升，因为 CRF 约束避免了部分错误标签组合）。

### 4. BIO 格式 vs Span 格式的工程适配

| 适配环节 | cluener (span) | 人民网 (BIO) |
|----------|:---|:---|
| 标签获取 | `text` → `span_to_bio()` → BIO ids | `ner_tags` → **直接查表** → BIO ids |
| 代码复杂度 | 需要 span 解析 + 边界处理 | **更简单**，一步到位 |
| 子词对齐 | `is_split_into_words=True` + `word_ids()` | **完全相同** |

人民网的 BIO 格式比 cluener 的 span 格式**更简洁**，数据加载路径缩短了 `span_to_bio()` 这一步。代码适配的核心就是新增一个 `PeoplesDailyDataset` 类，其余 tokenizer 处理、子词对齐、模型、训练循环全部复用。

### 5. max_length 的选择至关重要

人民网文本 P95=97 字，P100=577 字。选择 `max_length=256`：
- 覆盖 **99.9%+** 的样本（仅约 26 条被截断）
- GPU 显存友好（batch_size=32 时 RTX 4060 8GB 可正常运行）
- 远小于 BERT 512 上限，留有安全余量

若使用 cluener 的 `max_length=128`，将截断约 **1.7%** 的样本（~355 条）。

## 四、与 cluener 的完整对比总结

```
                      cluener (ner_demo)          人民网 (ner_demo2)
                      ─────────────────          ──────────────────
实体粒度               细粒度 (10类)               粗粒度 (3类)
标注格式               Span 格式                  BIO 格式
训练集规模             10,748                     20,864
测试集标签             ❌ 无                       ✅ 有
BERT+Linear F1 (val)   ~79.0%                     93.29%
BERT+CRF F1 (val)      ~72.5%                     93.70%
CRF vs Linear         -6.5% (CRF 更差)            +0.41% (CRF 略优)
最大文本长度            50 字                      574 字
推荐 max_length        128                        256
非法序列率 (Linear)    ~3-5%                      ~3.7%
```

### 教学启示

1. **数据集决定天花板**：同样的 BERT 架构，简单任务（3类）比复杂任务（10类）F1 高出 14 个百分点，说明任务本身的可分性是建模的第一决定因素。

2. **CRF 不是万能药**：在小数据集、短文本、类别多的场景下，CRF 可能反而损害性能（过拟合转移约束）；在大数据集、长文本、类别少的场景下，CRF 能稳定带来小幅度提升并消除非法序列。

3. **测试集有标签的价值**：cluener 的测试集无标签，无法评估最终泛化能力；人民网的测试集有标签，验证了模型从 93.29%（val）到 92.25%（test）仅有 1% 的泛化损失，证明 1 epoch 训练未过拟合。

4. **BIO 格式是 NER 的通用语言**：无论数据原始的标注格式是什么（span、CoNLL、JSON），最终都需要转换为 BIO 标签序列 + BERT 子词对齐。理解这一转换流程是 NER 工程的核心能力。

## 五、LLM API 评测（DeepSeek deepseek-v4-pro）

> 验证集随机采样 100 条，temperature=0.0，max_tokens=512

| 方案 | Precision | Recall | F1 | 说明 |
|------|:---:|:---:|:---:|------|
| **Zero-shot** | 82.43% | 57.37% | **67.65%** | 仅靠系统提示，无示例 |
| **Few-shot (3例)** | 80.61% | 49.53% | **61.36%** | 系统提示 + 3 个人民网标注示例 |

### 分析

1. **Zero-shot 反超 few-shot**（67.65% vs 61.36%）：few-shot 示例虽然提供了格式引导，但 3 个固定的示例可能限制了模型对多样实体的召回——recall 从 57.37% 降到 49.53%，说明模型过度模仿示例中的实体模式，漏掉了其他实体。

2. **Precision 高、Recall 低**：LLM 有较强的"不确定就不标"倾向——标出来的实体大多正确（82%），但大量实体被漏掉（recall 仅 57%）。这是 LLM NER 的典型模式。

3. **与 cluener 的对比**：cluener 上 Qwen API zero-shot F1 约 51.8%（10类），人民网上 DeepSeek zero-shot F1 67.65%（3类）——同样印证了"类别越少越容易"的规律。

## 六、LLM SFT 微调（Qwen2.5-0.5B-Instruct LoRA）

### 训练配置

| 参数 | 值 | 说明 |
|------|-----|------|
| 基座模型 | Qwen2.5-0.5B-Instruct | 495M 参数 |
| 微调方式 | LoRA (r=8, alpha=16) | 可训练 1.08M (0.22%) |
| 训练数据 | 人民网 20,864 条 | 全量训练集 |
| epochs | 3 | — |
| batch_size / grad_accum | 1 / 16 | 等效 batch=16 |
| max_length | 192 | 覆盖 P95+ |
| dtype | float16 | 适配 8GB 显存 |
| GPU 显存占用 | ~954MB | 远低于 8GB 上限 |
| 训练耗时 | ~94 分钟 | ~31 min/epoch |

### 训练过程

| Epoch | Val Loss | 耗时 |
|:---:|:---:|:---:|
| 1 | 0.0343 | 31.6 min |
| 2 | 0.0255 | 31.2 min |
| 3 | **0.0243** | 31.3 min |

> ⚠️ train_loss 因 float16 溢出显示为 NaN，但 val_loss 持续下降，模型质量未受影响。解决方案：改用 bf16 或启用梯度缩放。

### 评估结果（验证集随机采样 200 条）

| 指标 | 值 |
|------|:---:|
| Precision | 90.98% |
| Recall | 83.74% |
| **F1** | **87.21%** |
| JSON 解析失败率 | 0% (0/200) |

### 分析

1. **SFT 显著优于 LLM API**：87.21% vs 67.65%（zero-shot），提升 **+19.5pp**。即使是 0.5B 小模型，经过领域微调后远超通用大模型 API 的表现——验证了"小模型微调 > 大模型 zero-shot"的经典结论。

2. **低于 BERT+CRF 6.5pp**：87.21% vs 93.70%。差距来自两方面：
   - **生成式 vs 序列标注**：SFT 需要生成完整 JSON 再解析，BERT 直接逐 token 分类，天然更精确
   - **span 对齐误差**：SFT 用 `text.find()` 近似定位，同一实体多次出现时只取首位置；BERT 的 seqeval 严格按位置匹配

3. **JSON 输出完美合规**：0% 解析失败率，说明 3 epoch 训练已让模型完全学会 JSON 格式约束。

## 七、五方案完整对比

```
                         Precision   Recall     F1      显存     耗时
                         ─────────   ──────    ──────   ────    ────
BERT + Linear             92.35%     94.24%    93.29%   ~2GB    6.5min
BERT + CRF                93.16%     94.24%    93.70%   ~2GB    10min
Qwen2.5-0.5B LoRA SFT    90.98%     83.74%    87.21%   1GB     94min
DeepSeek zero-shot        82.43%     57.37%    67.65%   —       5min
DeepSeek few-shot         80.61%     49.53%    61.36%   —       8min
```

### 核心结论

1. **BERT 序列标注仍是中小规模 NER 的最优解**：93.70% F1，训练仅需 10 分钟（1 epoch），显存仅需 2GB。在 3 类实体任务上几乎达到实用上限。

2. **SFT 微调是 LLM 路线的正确打开方式**：从 zero-shot 的 67.65% 提升到 87.21%（+19.5pp），证明领域微调的价值。但性价比不如 BERT——耗时 15x，效果仍低 6.5pp。

3. **Zero-shot > few-shot 的反直觉现象**：精心挑选的 few-shot 示例反而损害了 recall（-7.8pp），提示在实际应用中使用 LLM API 做 NER 时，简洁的 zero-shot prompt 可能比复杂 few-shot 更有效。

4. **8GB 显存足够运行 0.5B 级别 LLM 的 LoRA 微调**：通过 float16 + batch_size=1 + grad_accum=16 的组合，实际显存占用仅 ~1GB，为笔记本场景下的 LLM 实验提供了可行的参数配置。

5. **BIO 格式适配的工程价值**：从 cluener（span）到人民网（BIO）的数据格式切换，核心改动仅 200 行代码（新增 `PeoplesDailyDataset` 类 + BIO 解析函数），其余模型/训练/评估代码全部复用。
