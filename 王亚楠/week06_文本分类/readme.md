文本分类从传统方法到LLM
文本分类应用场景
内容审核：（实时要求高）
- 垃圾评论检测、违规内容识别、风控文本分类
意图识别：（实时要求高）
- 对话系统 NLU、用户意图分类、槽位触发判断
情感分析：
- 商品评价分析、舆情监控分类、金融新闻情感
文本分类：
- 新闻自动打标、邮件智能归档、工单主题分类
客服质检：
- 问题类型识别、服务质量评估、投诉风险预警
共同特点：
- 类别预先定义，数量有限（几类 ~ 几百类）
- 大量标注数据可获取，支持监督学习
- 对推理延迟有要求，需轻量高效的部署方案
- 模型行为需可预期，便于线上审计和调试
传统文本分类方法
处理流程：原始文本——预处理——特征提取——分类器——类别标签
朴素贝叶斯
- 于贝叶斯定理 + 条件独立假设
- 训练快，小数据集上表现稳定
- 独立假设过强，忽略词序信息
支持向量机（SVM）
- 需要先生成词向量
Logistic 回归
- 线性分类器，输出概率可解释
- 可作为 BERT 分类头的基础形式
- 表达能力受限于线性决策边界
核心局限：
- 特征工程依赖强
- 语义理解能力弱
- 迁移能力差
仍有价值的场景：
- 极低资源的场景
- 强可解释性要求
- 快速冷启动验证业务可行性

训练目标要与使用方式一致
  - 为什么Bert最后分类要用cls向量进行，理论上来说每个token的向量都可以，这就是因为Bert的训练目标有nsp这个任务针对cls向量进行过训练，所以使用cls向量进行分类
微调工程实践要点：
学习率设置：
  - BERT 层：2e-5 ~ 5e-5（过大会破坏预训练知识）
  - 推荐使用 warmup + linear decay 调度
  - 可对 BERT 层和分类头分别设置不同 lr
训练超参：
  - Batch size：16 ~ 32（显存不足时用梯度累积）
  - 训练轮数：通常 3 ~ 5 epochs，过多易过拟合
  - 最长序列长度：128 或 512，取决于输入分布
常见工程坑：
  - 数据未 shuffle → 模型先见所有负样本后见正样本
  - 忘记 attention_mask → PAD token 参与了计算
  - tokenizer 截断长度与训练时不一致 → 线上效果差
防止过拟合：
  - Dropout（BERT 默认 0.1，不建议去掉）
  - 数据增强：回译、同义词替换、随机删除
  - 早停：验证集 F1 连续 N 轮不提升则停止
