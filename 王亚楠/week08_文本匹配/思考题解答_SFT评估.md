## 思考题 1：SFT 的 Accuracy 与 BERT 方案相比如何？用了多少训练数据？

**结论：SFT 全面落后于 12 层 BERT 判别式方法，但数据用量远少于 BERT。**

| 数据集 | 最佳 BERT F1 | SFT F1 (正例) | 差距 | BERT 训练数据 | SFT 训练数据 |
|--------|:-----------:|:------------:|:----:|:----------:|:----------:|
| BQ_Corpus | 0.9349 (CrossEncoder) | 0.8152 | **−11.97** | 68,960 条 | **5,000 条** (1/14) |
| LCQMC | 0.8946 (CrossEncoder) | 0.8543 | **−4.03** | 238,766 条 | **5,000 条** (1/48) |
| AFQMC | 0.6765 (BiEncoder, 4层) | 0.5556 | −12.09 | 34,334 条 | **5,000 条** (1/7) |

### 关键分析

**SFT 训练数据少得多**：两个数据集 SFT 都只用 5,000 条（正负各半的平衡采样），而 BERT 使用了全部训练数据（BQ 69K、LCQMC 239K）。SFT 用了 **1/14 到 1/48** 的数据量。

**SFT 的精度差距在缩小**：LCQMC 上 SFT 与 BERT 仅差 4 个点（0.8543 vs 0.8946），比 BQ_Corpus 的 12 个点差距小得多。开放域数据的语义多样性让生成式的灵活表达能力发挥出来，而金融领域的模板化表述更适合判别式建模。

**SFT 的真正价值不在精度**：从精度看 SFT 确实落后，但它的核心价值是：

1. 无需标注数据也能用 LLM 做匹配（zero-shot 在 LCQMC 上已达 0.8571）
2. 可灵活扩展新类别（如三分类"相似/不相似/部分相似"），判别式需要改架构重训练
3. 离线本地部署，无 API 成本
4. 同一模型可同时做匹配 + 解释 reasoning

## 思考题 2：生成式方法 parse_fail 率有多高？与 NER 任务相比为什么更低？

**结论：文本匹配 parse_fail 率为 0%，远优于 NER 任务。**

| 数据集 | SFT parse_fail | LLM Zero-Shot parse_fail |
|--------|:-------------:|:------------------------:|
| BQ_Corpus | **0%** (0/200) | **0%** (0/100) |
| LCQMC | **0%** (0/200) | **0%** (0/100) |
| AFQMC | **0%** (0/200) | 未统计 |

### 为什么比 NER 低得多？

| 维度 | 文本匹配 | NER |
|------|---------|-----|
| TARGET 长度 | **3-5 token**（【相似】/【不相似】） | **20-150 token**（完整 JSON） |
| 输出空间 | **闭集二选一**，只有两个合法答案 | **开放集**，实体名、类型、位置自由组合 |
| 格式约束 | 只需输出的文本包含"相似"/"不相似"即可 | 需要严格遵守 JSON schema，字段名、引号、括号、逗号都不可出错 |
| 模型学到的方式 | 分类记忆——两个标签在训练中出现数千次，模型很快建立"这个问题格式 → 这两个输出之一"的映射 | 结构化生成——需要同时学会识别实体 + 生成正确 JSON 格式，两件事耦合在一起 |
| 解析容错 | 极高——只要输出中含"相似"或"不相似"就能正确解析 | 极低——JSON 少一个括号或字段名拼错就无法解析 |

**核心原因**：文本匹配本质上是**把二分类问题伪装成了生成任务**。TARGET 极短且固定，训练时模型很快学到输出只有两种可能，不会产生格式错误。而 NER 的 JSON 输出长度长、结构复杂，每一步生成都可能偏离 schema。这就是为什么文本匹配是所有 LLM SFT 任务中 **parse_fail 趋近于零** 的任务。

## 思考题 3：BERT BiEncoder 可以做向量检索，SFT 方法可以吗？各自适合什么场景？

**结论：SFT（LLM 生成式）不能直接做向量检索。两者是互补关系，而非替代关系。**

### BiEncoder 的核心能力——向量检索

```
encode("花呗怎么还款") → [0.12, -0.34, 0.56, ..., 0.78]  # 768 维 L2 归一化向量
                          ↓
                    FAISS IndexFlatIP
                          ↓
          从 100 万候选问句中召回 Top-100 相似句（毫秒级）
```

BiEncoder 的 `encode()` 输出是固定维度的向量，与 FAISS/Milvus 等向量数据库直接集成，支持**亿级候选的毫秒级检索**。

### SFT 方法为什么不能

LLM 是 token-by-token 生成的，没有"把整个句子压缩为固定向量"的机制。每次推理都要走完整前向传播 + 生成过程，无法预计算、无法建立向量索引。

### 能力对比矩阵

| 能力 | BiEncoder | CrossEncoder | LLM SFT |
|------|:---:|:---:|:---:|
| 向量检索（大规模召回） | ✅ 原生支持 | ❌ 需逐对计算 | ❌ 无法向量化 |
| 精度（小规模精排） | 中等 | **最高** | 较低 |
| 离线预计算 | ✅ 句子向量可预存 | ❌ 每次重算 | ❌ 每次生成 |
| 灵活扩展（多类别/解释） | ❌ 只输出相似度 | ❌ 只输出 0/1 | ✅ 可输出任意格式 |
| 本地离线推理 | ✅ ~5ms | ✅ ~20ms | ✅ ~1s（更慢） |
| 适合场景 | RAG Recall 阶段 | RAG Rerank 精排 | 需要解释/多分类/零样本冷启动 |

### 最佳实践：级联架构

```
用户查询 → BiEncoder (FAISS 向量召回 Top-100)
             ↓
         CrossEncoder (精排 Top-10)
             ↓  (可选)
         LLM SFT (对 Top-3 做最终判断 + 输出解释)
```

这就是 RAG 系统中经典的 **Recall → Rerank** 两阶段流水线，LLM 在最后一环做高价值但低频的深度判断。

## 思考题 4：文本匹配的 SFT TARGET 只有 2~3 个 token，与 NER（20~150 token）相比训练有什么不同？

**结论：文本匹配的 SFT 学习信号极度集中，训练快但更容易过拟合；NER 学习信号分散，需要更多数据和 epoch 才能收敛。**

| 维度 | 文本匹配 SFT | NER SFT |
|------|:---:|:---:|
| TARGET token 数 | 3-5（【相似】/【不相似】） | 20-150（完整 JSON） |
| max_length 建议 | **128** | **256** |
| loss 集中程度 | **极高**（几乎全部集中在最后 3-5 个 token） | 分散（loss 分布在所有实体 token 上） |

### 具体影响

#### 1. 收敛速度

```
文本匹配 SFT 训练过程:
  Epoch 1: train_loss=0.088, val_loss=0.079  ← 已经收敛！
  Epoch 2: train_loss=0.051, val_loss=0.100  ← 开始过拟合
  Epoch 5: train_loss=0.008, val_loss=0.171  ← train_loss 接近 0，val_loss 翻倍

NER SFT 训练过程（参照项目经验）:
  Epoch 1: train_loss=0.35, val_loss 缓慢下降
  Epoch 3: 才基本收敛
  Epoch 5: 可能还在继续提升
```

文本匹配 **1 epoch 已收敛**，2+ epoch 必然过拟合。TARGET 太短，模型学得太快。

#### 2. 梯度更新范围

文本匹配的 loss 几乎全部来自最后几个 token 的生成（"相"、"似"/"不"、"相"、"似"），前面的 context token（prompt 部分）梯度贡献接近于零。这意味着：

- **有效的反向传播路径极短**（从最后 token 到输出头，只有几层 decoder layer 的贡献）
- **模型容量利用不充分**——495M 参数，但只有输出 head 附近的参数真正参与了"选择相似还是不相似"的决策

NER 的每个实体 token 都产生 loss，因此梯度从 JSON 的各个位置传导回来，覆盖更多参数。

#### 3. 平衡采样的重要性

由于 loss 集中在极少数 token 上，类别不均衡（如 AFQMC 69% 负例）会迅速被模型捕捉——"一直输出【不相似】"就能获得约 69% 的 token-level accuracy。因此 **平衡采样是必须的**，否则模型退化速度比 NER 快得多（NER 的 loss 分布在整个序列，单一位置的错误影响被稀释了）。

#### 4. 推理效率

文本匹配的推理效率更高：

- 文本匹配：`max_new_tokens=8`（实际只用 3-5）
- NER：`max_new_tokens=256`
- 解码时间：文本匹配 ~1s/条，NER ~5-10s/条

从效率角度看，文本匹配是更适合 SFT 方案的任务。

## 总结

| 维度 | 文本匹配 SFT | NER SFT | 分类 SFT |
|------|:---:|:---:|:---:|
| TARGET token | 3-5 | 20-150 | 1-2 |
| 收敛所需 epoch | **1** | 3-5 | 1-2 |
| parse_fail 风险 | **极低** | 中等 | **极低** |
| 过拟合风险 | **高**（TARGET 太短） | 低（信号分散） | 中 |
| 平衡采样必要性 | **必须** | 建议 | **必须** |
| 适合 LLM SFT 程度 | ⭐⭐⭐⭐ | ⭐⭐ | ⭐⭐⭐⭐⭐ |

文本匹配四个任务特点——TARGET 短、输出封闭、格式容错高、收敛快——让它成为 LLM SFT 教学中承上启下的理想案例：比分类任务多了一层"解析生成输出"的工程挑战，又没有 NER 的 JSON 格式复杂度，适合在三类任务中作为**从分类到 NER 的过渡**。
