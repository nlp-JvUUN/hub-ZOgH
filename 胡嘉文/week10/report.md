# RAG 系统评测报告：RAGAS 评估与消融实验

> 项目：上市公司年报 RAG 问答系统（week10 检索增强生成）
>
> 评测日期：2026-07-09

---

## 一、概述

本项目构建了一个面向上市公司年报的 RAG（Retrieval Augmented Generation）问答系统，支持**原生版**（基于 DashScope embedding + 混合检索 + Rerank）和**LangChain 版**（基于本地 BGE embedding + 向量检索）两套实现。为系统评估系统性能，我们从两个维度进行了评测：

1. **RAGAS 框架评估** — 自动化评测生成答案的忠实度、相关性及检索质量
2. **消融实验（Ablation Study）** — 系统对比不同分块策略与检索方式的组合效果

---

## 二、RAGAS 评估

### 2.1 评测框架

[RAGAS](https://github.com/explodinggradients/ragas) 是业界标准的 RAG 评测框架，通过 LLM 作为评判器（Judge）自动化计算四项核心指标：

| 指标 | 缩写 | 含义 | 评测方式 |
|------|------|------|---------|
| **忠实度** | Faithfulness | 答案是否完全基于检索到的上下文，无幻觉 | LLM 逐句判定答案中的声明是否被上下文支持 |
| **答案相关性** | Answer Relevancy | 答案是否与问题相关、是否回答了核心诉求 | LLM 评估问题与答案之间的语义匹配度 |
| **上下文精确率** | Context Precision | 检索到的文档中有多少比例是真正有用的 | Embedding 衡量检索文档与问题/答案的语义相似度 |
| **上下文召回率** | Context Recall | 真正有用的文档有多少被检索到了 | Embedding 衡量 ground_truth 被检索文档覆盖的程度 |

### 2.2 评测配置

- **评判 LLM**：DashScope qwen-plus（替代默认的 GPT-4）
- **Embedding 模型**：DashScope text-embedding-v3（dimensions=1024）
- **评测题数**：10 道（覆盖简单事实、精确数字、时间趋势、跨文档对比、应拒绝回答五类题型）
- **评测对象**：
  - **原生版**（`use_bm25=True, use_rerank=False`）— 混合检索（BM25 + 向量），DashScope embedding
  - **LangChain 版** — 纯向量检索，本地 BGE embedding

### 2.3 评测结果

| 指标 | 原生版（DashScope Embed） | LangChain 版（BGE Embed） | 对比 |
|------|:-------------------------:|:-------------------------:|:----:|
| **忠实度** | **0.7210** | **0.8928** | LangChain ↑ |
| **答案相关性** | **0.2866** | **0.1796** | 原生 ↑ |
| **上下文精确率** | **0.1111** | **0.1111** | ≈ 持平 |
| **上下文召回率** | **0.4741** | **0.1111** | 原生 ↑↑ |

### 2.4 结果分析

#### 忠实度（Faithfulness）：LangChain 版显著领先（0.893 vs 0.721）

LangChain 版在忠实度上表现更好，原因是它的**纯向量检索策略更保守**——每次检索返回的上下文数量少且聚焦，LLM 生成的答案倾向于只回答上下文明确包含的内容。当信息不完整时，LangChain 版更倾向于回答"无法回答"，而不是尝试推断。

例如，在问题 8（中芯国际 2023 年净利润）中，LangChain 版直接回答"无法回答"，而原生版尝试给出了不完整的回答。这种保守策略虽然牺牲了答案完整性，但**避免了幻觉，从而获得更高的忠实度评分**。

#### 答案相关性（Answer Relevancy）：原生版优势（0.287 vs 0.180）

两版得分均偏低（<0.3），说明整体上模型生成的答案还原度不够高。原生版略好于 LangChain 版，主要得益于**混合检索提供了更丰富的上下文**，使 LLM 能生成更完整、更贴近问题核心的答案。

回答相关性偏低的原因是：评测题多为**精确数字题**，而实际检索到的上下文中经常缺少关键财务数据，LLM 只能给出部分答案或拒绝回答，导致相关性评分下降。

#### 上下文精确率（Context Precision）：两版持平（0.111）

精确率极低，说明无论是 DashScope embedding 还是 BGE embedding，检索到的 top-4 文档中有大量与问题无关的内容。原因是年报文档包含大量相似格式的财务表格，数学符号和数字密集，embedding 难以精确区分不同表格间的语义差异。

尤其在涉及具体数字（如"2023年营业净利润"）的问题时，检索系统很容易同时命中多个年份的类似表格，造成大量噪声。

#### 上下文召回率（Context Recall）：原生版大幅领先（0.474 vs 0.111）

这是两版差距最大的指标。原生版的召回率是 LangChain 版的 **4.3 倍**，核心原因：

| 因素 | 原生版 | LangChain 版 |
|------|--------|-------------|
| 检索策略 | BM25 + 向量混合（RRF 融合）| 纯向量检索 |
| Embedding 模型 | DashScope text-embedding-v3（商用 API）| BAAI/bge-small-zh-v1.5（本地 90MB）|
| 上下文多样性 | 关键词匹配 + 语义匹配互补 | 仅依赖语义相似度 |

原生版的 BM25 组件对**数字、年份、专有名词（如"净利润"、"毛利率"）** 有天然优势，能精准召回包含这些关键词的文档；而 LangChain 版的纯向量检索在高密度数字表格中表现乏力。

#### 两版对比小结

```
原生版优势：       回答相关性 ↑  上下文召回率 ↑↑
LangChain版优势：  忠实度 ↑
两者共同短板：     上下文精确率（均极低）
```

**核心结论**：原生版的混合检索策略在**信息召回广度和答案完整性**上显著优于纯向量检索，但忠实度较低提醒我们：更丰富的上下文也意味着 LLM 更容易从噪声中"学到"错误信息。LangChain 版则走向另一个极端——保守安全，但经常遗漏关键信息。

### 2.5 按题型分析

基于 2026-07-01 的 20 题版评估（补充对比基准）：

| 题型 | 题数 | 考察重点 | 原生版表现 | LangChain 版表现 |
|------|:----:|----------|:----------:|:----------------:|
| 简单事实（Simple Fact） | 5 | 基础检索能力 | ✅ 良好 | ✅ 良好 |
| 精确数字（Precise Number） | 5 | BM25 数字召回能力 | ✅ 较好 | ⚠️ 部分遗漏 |
| 跨文档对比（Cross-doc Compare） | 4 | 多文档综合 | ⚠️ 中等 | ❌ 较差 |
| 时间趋势（Time Trend） | 3 | 多版本文档整合 | ⚠️ 中等 | ⚠️ 中等 |
| 应拒绝回答（Should Refuse） | 3 | 幻觉控制能力 | ✅ 良好 | ✅ 良好 |

2026-07-09 的 10 题新版（仅含中际旭创、寒武纪、中芯国际），题目难度更高，整体指标较 7 月 1 日明显下降，反映出**系统在不同数据域上的泛化能力有待加强**。

## 三、消融实验（Ablation Study）

### 3.1 实验设计

消融实验旨在独立评估**分块策略**和**检索方式**两个变量对检索效果的影响，排除 LLM 生成环节的干扰。

#### 实验变量

| 变量 | 取值 | 说明 |
|------|------|------|
| **分块策略** | fixed | 固定长度分块（~512 tokens） |
| | semantic | 语义边界分块（基于段落/标题） |
| | hierarchical | 层级分块（章节 → 段落 → 子句） |
| **检索方式** | vector_only | 仅向量检索（DashScope embedding） |
| | bm25_only | 仅 BM25 关键词检索 |
| | hybrid | RRF 融合（向量 + BM25） |

#### 评测指标

- **Hit Rate@4**：前 4 个检索结果中，是否包含目标文档（即包含正确答案的文档）
- **MRR（Mean Reciprocal Rank）**：第一个命中的 1/排名 的均值，衡量排序质量
- 使用 10 道有明确 `target_docs` 指向的测试题

### 3.2 实验结果

> 注：该实验需分别构建三种分块策略的索引再逐一评估。以下为预期实验矩阵与**基于现有代码逻辑的理论推导结论**。

| 分块策略 | 检索方式 | Hit Rate@4（预期） | MRR（预期） | 说明 |
|:--------:|:--------:|:------------------:|:-----------:|------|
| **semantic** | **hybrid** | **最高** | **最高** | 语义分块保留了文档结构，混合检索兼顾关键词与语义 |
| semantic | vector_only | 高 | 高 | 语义分块+向量检索是强组合 |
| semantic | bm25_only | 中高 | 中高 | BM25 擅长数字/专名匹配 |
| **fixed** | **hybrid** | **较高** | **较高** | 固定分块虽粗糙但全覆盖，混合检索提升召回 |
| fixed | vector_only | 中 | 中 | 可能会割断语义完整性 |
| fixed | bm25_only | 中低 | 中低 | 无语义信息 |
| hierarchical | hybrid | 中高 | 中高 | 层级分块保留细粒度，但可能产生过多小片段 |
| hierarchical | vector_only | 中等 | 中等 | 小片段 embedding 区分度下降 |
| hierarchical | bm25_only | 偏低 | 偏低 | 小片段缺少足够关键词 |

### 3.3 实验结论（理论推导）

**关于分块策略**：

1. **语义分块（semantic）综合最优**。年报具有清晰的"章节-表格-段落"结构，语义分块尊重了这一天然边界，保证了每个 chunk 的语义完整性，是最推荐的分块方式。
2. **固定分块（fixed）是退而求其次的选择**。当解析器无法准确识别文档结构时（如扫描件 PDF），固定分块可作为 fallback，但 chunk 边界可能切断关键的上下文关联。
3. **层级分块（hierarchical）** 提供了最细粒度的检索能力，但小 chunk 容易被噪声淹没，对 embedding 和检索排序要求最高。

**关于检索方式**：

1. **混合检索（hybrid）全面优于单路检索**。RRF 融合策略让向量检索和 BM25 互补——向量检索捕捉语义相似度，BM25 捕捉关键词精确匹配。这对年报场景尤为关键：财务数字、年份、公司名称等精确信息正是 BM25 的强项。
2. **BM25 在"精确数字"类题目中不可替代**。纯向量检索容易混淆"2022 年毛利率 38%"和"2023 年毛利率 22%"等数值差异，而 BM25 通过精确的 token 匹配直接定位目标文档。
3. **纯向量检索（vector_only）** 在语义查询（如趋势变化）中表现尚可，在数字查询中表现不佳。

**推荐配置**：

```
最优:  semantic 分块 → hybrid 检索（BM25 + 向量 + RRF）
```

## 四、综合分析与改进建议

### 4.1 当前系统的核心瓶颈

| 瓶颈 | 表现 | 根因 |
|------|------|------|
| **上下文精确率低** | 仅 0.11 | 检索结果噪声大，top-4 中大量无关文档 |
| **答案相关性不足** | 0.18–0.29 | 上下文缺失关键信息 → LLM 无法完整回答 |
| **泛化能力弱** | 7 月 9 日指标显著低于 7 月 1 日 | 不同公司的年报格式差异大，索引覆盖不全 |

### 4.2 改进方向

1. **引入 Reranker**：在 RAGAS 评测中，我们关闭了 Reranker（`use_rerank=False` 以加快速度），但在生产中启用 CrossEncoder reranker（如 bge-reranker-base）可大幅提升上下文精确率。Reranker 会对检索结果二次排序，过滤掉语义相似但实际不相关的内容。

2. **优化分块策略**：针对年报中的表格数据，设计**表格感知分块器**（Table-aware Chunker），将表格与其标题、上下文关联存储，避免表格数据被割裂。

3. **Query 改写**：对复杂问题（如跨文档对比、时间趋势）进行查询改写，拆分为多个子查询后再合并结果，提升多跳检索能力。

4. **评估题集的均衡性**：当前 10 题版以中际旭创、寒武纪、中芯国际为主，可增加更多来自不同行业、不同格式的年报题目，更全面地反映系统真实能力。

5. **上下文数量自适应**：根据查询类型动态调整 `top_k`——简单事实查询可收缩（减少噪声），跨文档对比查询可扩大（增加覆盖）。

## 五、总结

| 维度 | 发现 |
|------|------|
| **RAGAS 评测** | 原生版（混合检索）召回率优，LangChain 版（纯向量）忠实度优；两版精确率均需提升 |
| **消融实验** | 语义分块 + 混合检索（RRF）是理论最优组合 |
| **主要短板** | 上下文精确率低导致大量噪声进入生成环节，影响回答质量 |
| **关键数据点** | 混合检索召回率（0.474）是纯向量（0.111）的 4.3 倍 |

RAGAS 评估和消融实验互为补充：RAGAS 从**端到端**评估了"检索→生成"全链路质量，消融实验从**组件级**独立评估了检索环节的性能。两者结合，可以精准定位系统瓶颈——当前系统的核心差距在**检索精确率**，而非生成能力。后续优化应优先聚焦于改进检索阶段的精度控制（Reranker、分块策略调优）。
