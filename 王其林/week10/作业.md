# 人民日报 RAG 系统实验报告

> 项目：rag_excise 检索增强生成系统实战
> 数据：人民日报 2026-06-01 ~ 2026-07-05（共 35 天 PDF）
> 生成时间：2026-07-09

---

## 一、实验目标

1. 构建完整的人民日报 RAG 检索增强生成系统（PDF 解析 → 分块 → 索引 → 检索 → 问答）。
2. 对比 **三种分块策略**（固定大小 / 语义分块 / 层级父子块）在三类检索模式（向量 / BM25 / 混合）下的效果。
3. 评估在官方提供的 50 道题目集上的命中率（Hit@4）、排序质量（MRR）与上下文长度（AvgLen）。
4. 分析各策略的优劣与互补性，给出推荐的默认配置。

*注：项目中已实现的实体抽取、关系抽取、知识图谱增强代码保留，但本次实验聚焦分块策略与检索模式的对比，未将图谱功能纳入主评估流程。*

---

## 二、技术选型

### 2.1 整体架构

项目采用典型的 **RAG 离线-在线两阶段架构**：

```
PDF → 解析(PDF→JSON) → 分块(3策略) → 索引(FAISS向量 + BM25倒排)
                                              ↓
用户查询 → 混合检索(向量+BM25) → RRF融合 → 时序加权排序 → Rerank → 上下文 → LLM/演示
```

### 2.2 各模块技术栈

| 模块 | 选型 | 版本 / 参数 | 说明 |
| :--- | :--- | :--- | :--- |
| **PDF 解析** | pdfplumber + PyMuPDF (fitz) | pdfplumber 0.11.10 / fitz 1.28.0 | 双引擎混合：pdfplumber 负责版面与表格提取，PyMuPDF 提供高效文本块定位；预留 OCR 接口（is_ocr 标记）。 |
| **分块策略 A** | 固定大小分块 (fixed) | chunk=500字，overlap=50字 | 基线策略：朴素滑窗，块大小固定。 |
| **分块策略 B** | 语义分块 (semantic, **默认**) | max=800字，min=80字 | 基于解析结构分块：遇 `block_type=title` 或表格强制切分，保证语义完整性。 |
| **分块策略 C** | 层级父子分块 (hierarchical) | parent=2000字，child=400字，overlap=50 | 长父块供 LLM 阅读，短子块供向量检索；理论上兼顾召回与阅读连贯性。 |
| **Embedding 模型** | BAAI/bge-large-zh-v1.5 | 本地离线，1024 维 | 中文通用语义表征的业界强基线。 |
| **向量索引** | FAISS (IndexFlatIP，内积) | faiss-cpu 1.14.3 | 精确内积检索，支持 macOS Apple Silicon MPS 加速 embedding 计算。 |
| **关键词检索** | BM25 (rank_bm25 + jieba) | jieba 0.42.1 | jieba 中文分词 + BM25 Okapi 打分，提供与向量互补的字面匹配信号。 |
| **融合策略** | 倒数秩融合 (RRF, Reciprocal Rank Fusion) | `k=60` | 无权重超参，简单稳健地融合向量 Top-K 与 BM25 Top-K 的排序结果。 |
| **时序加权** | 时间衰减加分排序 | `decay_factor=0.995` | 越新的文档获得越高加分：`final_score = rrf_score × decay_factor^days_ago`，符合新闻类数据的时效性特征。 |
| **Rerank 重排** | BAAI/bge-reranker-base | 本地 CrossEncoder | 对 RRF+时序 的 Top-K 结果用 CrossEncoder 做精细重排（默认 20→6）。 |
| **LLM 问答** | DeepSeek API（预留） | DEEPSEEK_API_KEY | 生成式回答，本次作业为保证可复现与离线演示，采用「Top-N 检索上下文拼接」作为可解释的答案演示。 |
| **评估框架** | RAGAS（预留）+ 自研脚本 | Hit@4 / MRR / AvgLen | RAGAS（Faithfulness / Relevancy / Context Precision 等）需要 LLM API，本次报告核心用自研指标衡量检索层效果。 |

## 六、整体分析与讨论

### 6.1 为什么 Fixed + Hybrid 反而夺冠？

这是本次实验最反直觉的结论之一。我们分析原因有三：

1. **块大小更均衡**：Fixed 强制 500 字 + 50 重叠，BGE 模型对「中等长度中文」的表征最稳定——而 Semantic 切出的 2 字短块（比如纯标题行）虽然保留了结构，但向量表征极其稀薄。
2. **新闻文本的特殊性**：人民日报是「高密度结构化政经文本」，标题+正文通常在同一块内表达完整语义，500 字刚好容纳一小节。
3. **Hybrid = 优势双保险**：RRF 融合既拿到了 Fixed 向量召回的高下限（0.778），又加上 BM25 对精确数字/日期的强字面匹配，一举从 0.778 → 0.944，+16.6 pp。

### 6.2 Semantic 真的落后吗？

**并不。** Semantic + Hybrid 仍有 0.889 Hit@4，同时它额外带来了两个不可替代的优势：

1. **文章标题信息零丢失**：Semantic 的每个 chunk 都携带 `article_title`（5,706 篇全覆盖），这对「溯源展示」和「去重合并」非常重要。Fixed 不保留标题，LLM 看到一段 500 字但不知道文章标题，难以做引用链接。
2. **按结构切分更符合 LLM 阅读**：表格、标题、列表都被妥善完整保留，不容易出现把一条表格记录从中间切断的情况。

**因此工程建议**：在离线流程中同时建立 Semantic 索引（用于检索+溯源展示）和 Fixed 索引（作为备用召回层），或用 Semantic 的块但在 embedding 时拼接 `article_title + " | " + content` 来补齐短标题块的向量表征差距。

### 6.3 层级分块的「名不副实」与使用建议

Hierarchical 在本次实验中：
- BM25 召回达到 **0.944**（全局第一梯队）
- 但向量召回仅 **0.667**，拖了 Hybrid 的后腿

原因可能是：
1. child=400 字略短，新闻中「时间+数字+人物」三元组恰好被跨块切断，导致向量检索召回下降。
2. 本次演示最后只用 Rerank，没有实现「检索 child → 返回 parent chunk 给 LLM」的完整链路（代码里保留了 parent 元数据），**层级的真正优势（给 LLM 更完整上下文）尚未被 RAGAS 的 Faithfulness 指标验证**。

**建议**：如果后续接入 LLM 并评测「回答忠实度 / 完整度」（Faithfulness / Answer Completeness），再把 hierarchical 作为主策略重新评估，届时它的收益会体现。

### 6.4 混合检索 (Hybrid) 的必要性

从 5.4 节的「互补性」可以看到：三种策略中，**Hybrid 都修复了单模 5~6 题的失败**。对新闻类数据而言：
- 向量召回擅长：同义改写、模糊描述、跨表述回忆（如「红领巾讲解员」→「少先队讲解员」）
- BM25 擅长：专有名词精确命中、数字、日期、政策文号、人名、地名

两者是典型的 **正交信号**。结合本次数据看：**任何一种策略下，Hybrid 都 ≥ max(vector, bm25) 的 Hit@4**，RRF 融合不会减分。

### 6.5 时间衰减加分的作用

本次实验中混合检索默认开启 `decay_factor = 0.995` 的时间衰减：
- 对 time_trend 类问题，Fixed/Hybrid 的 MRR 从纯 BM25 的 0.576 提升到 **0.796**（+22 pp），是全实验中 MRR 提升最显著的一处。
- 因为新闻问题（如「最新的就业数据」「最近的销量」）天然偏重新近信息，时间加权在排序阶段的贡献不可替代。

### 6.6 局限性与改进方向

| 方向 | 现状 | 改进建议 |
| :--- | :--- | :--- |
| **LLM 层评估缺失** | 本次报告聚焦检索层。RAGAS 因依赖 LLM API 未完整跑通（Faithfulness / Answer Relevancy 空缺）。 | 接入 DeepSeek 等 LLM 后，跑 RAGAS 全量 50 题，补完生成侧指标。 |
| **知识库时间窗** | 仅 35 天（6/1~7/5），50 道题中只有 18 道在范围内。 | 扩展到 6 个月或全年，50 道题全部覆盖，结论更稳。 |
| **Embedding 中文适配** | BGE-large-zh-v1.5 是通用中文模型，可能未针对新闻体裁微调。 | 尝试「BGE-M3 / GTE-Qwen / 基于人民日报增量微调」提升召回。 |
| **Rerank 对 latency 影响** | 当前单问 2.6s，Rerank 占 70%+。 | 先用向量+BM25 粗排 Top8，Rerank 只在交互式场景开启。离线批量评估时可关闭。 |
| **Semantic 短块向量退化** | 极短 chunk（2 字标题行）向量失真。 | 在 embedding 阶段拼接 `{article_title} [SEP] {content}` 并按字符上限截断。 |
| **层级分块 parent 返回** | 本次 child 召回后未换 parent 送 Rerank。 | 新增「child→parent 还原」链路，用 2000 字父块作为实际上下文送 LLM。 |

## 七、结论与推荐

### 7.1 推荐配置

| 优先级 | 组合 | Hit@4 | MRR | 推荐场景 |
| ---: | :--- | ---: | ---: | :--- |
| 🌟 **默认首选** | **fixed + hybrid** | **0.944** | **0.749** | 离线批量问答、题库命中最大化、对命中率要求最高的场景。 |
| 🌟 生产首选 | semantic + hybrid | 0.889 | 0.668 | 需要溯源展示、文章标题清晰、面向 C 端交互的 RAG 产品。 |
| 研究方向 | hierarchical + bm25_only / hybrid | 0.944 | 0.719 / 0.697 | 后续接入 LLM 后，考察 Faithfulness 的实验方向。 |

### 7.2 工程落地建议清单

1. **检索层必做**：**向量 + BM25 + RRF 融合 + 时间衰减排序** 的四件套必须全部开启，缺一不可。
2. **分块策略**：优先 **fixed + semantic 双索引** 并行，semantic 供溯源展示，两者 RRF 结果取并集。
3. **延迟优化**：Rerank 在高并发场景可降级，Hit@4 从 0.944 下降有限，但吞吐提升 3~4 倍。
4. **时序数据**：时间衰减是新闻类 RAG 的「免费午餐」，建议对「时事 / 政务 / 新闻」类语料默认开启。
5. **拒绝类问题**：配合 `should_refuse` 题型 + 检索置信度阈值（如 Top1 相似度＜某值即拒绝），可避免 LLM 幻觉式回答「资料外事实」。
6. **扩展评估**：一旦接入 LLM，用 RAGAS 在全量 50 题上补充 Faithfulness / Context Precision / Answer Relevancy，形成「检索 + 生成」双视角指标。

### 8.7 总结与结论

1. **命中率**：Native **88.9% (16/18)** vs LangChain **83.3% (15/18)**，Δ = −5.6pp → **Native 胜**（差 1 题属正常波动）。
2. **排序质量（MRR）**：Native **0.696** vs LangChain **0.750**，Δ = +0.054 → **LangChain 把黄金文档排到 Top1 的比例更高**。
3. **工程耗时**：检索端 2313ms vs **1812ms**（−501ms，−21%），LLM 端 7881ms vs **5457ms**（−2424ms，−31%）。LangChain 封装未引入延迟损失，反而因 semantic 分块上下文更紧凑，**双端都更快 21%~31%**。
4. **答案安全性**：两版拒绝率均 **55.6%**（10/18 题属语料外），**均无幻觉瞎编**，RAG「约束回答范围」目标达成。
5. **题型级建议**：① 业务以 **跨文档对比** 为主 → 优先 LangChain semantic（+25pp）；② 业务以 **精确数字抽提** 为主 → 优先 Native fixed（+20pp）；③ 折中方案：两套索引做 RRF 融合。
6. **可替代性判断**：**LangChain 版与 Native 版在命中率、MRR、工程耗时上达到同一水平，可互相替代**。需切换 Vector Store / Provider 时选 LangChain；追求极致可调试性选 Native。
