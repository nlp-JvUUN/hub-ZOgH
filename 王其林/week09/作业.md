# vLLM 部署与约束解码实验报告

> **实验环境**：Mac M4 (Apple Silicon) / 32GB 统一内存 / macOS
> **推理引擎**：vllm-metal v0.3.0（vLLM 0.23.0 + Metal 后端 + xgrammar FSM）
> **原版参考**：NVIDIA RTX 4060 8GB / WSL2 Ubuntu 22.04 / vLLM 0.9.2 + CUDA 12.7
> **实验模型**：Qwen2.5-0.5B-Instruct

---

## 一、实验背景

本项目通过 **6 个脚本** 演示 vLLM 推理框架的 3 大核心能力：

| 教学目的 | 对应脚本 | 核心问题 |
|---------|---------|---------|
| **部署能力** | `start_server.sh` | 如何将 HuggingFace 模型一键变为 OpenAI 兼容 HTTP 服务？ |
| **性能能力** | `bench_throughput.py` | 为什么 vLLM 比原生 transformers 快几十倍？ |
| **约束解码能力** | `demo_guided_*.py` + `demo_function_call.py` | 如何保证 LLM 输出的 JSON 100% 符合 Schema？ |

通过 **vllm-metal 插件** 将原版 NVIDIA CUDA 项目完整移植到 Mac M4。

---

## 二、架构适配

### 2.1 硬件差异

| 组件 | 原版 (NVIDIA) | Mac (vllm-metal) |
|------|--------------|------------------|
| GPU | CUDA 8GB | Metal 3 统一内存 32GB |
| vLLM | 0.9.2 CUDA kernel | 0.23.0 Metal kernel |
| 约束解码 | xgrammar FSM (CUDA) | xgrammar FSM (Metal) |
| 显存管理 | PagedAttention (CUDA) | PagedAttention (Metal) |
| Continuous Batching | ✅ | ✅ |

### 2.2 API 参数适配（关键）

vLLM 0.23.0 将约束解码参数从 `guided_*` 改为 `structured_outputs`，旧参数被**静默忽略**（不报错）：

| 约束类型 | 旧参数 (vLLM < 0.23) | 新参数 (vLLM 0.23+) |
|---------|---------------------|---------------------|
| JSON Schema | `extra_body={'guided_json': s}` | `extra_body={'structured_outputs': {'json': s}}` |
| 枚举选择 | `extra_body={'guided_choice': c}` | `extra_body={'structured_outputs': {'choice': c}}` |
| 正则 | `extra_body={'guided_regex': r}` | `extra_body={'structured_outputs': {'regex': r}}` |

### 2.3 启动命令

```bash
curl -fsSL https://raw.githubusercontent.com/vllm-project/vllm-metal/main/install.sh | bash
source ~/.venv-vllm-metal/bin/activate
vllm serve pretrain_models/Qwen2.5-0.5B-Instruct \
    --served-model-name qwen2.5-0.5b --host 0.0.0.0 --port 8000
```

## 三、实验结果

### 3.1 吞吐对比（50 条 prompt × 100 token）

#### Mac M4 (vllm-metal)

| 模式 | 总耗时 | QPS | tok/s | 相对 vLLM |
|------|--------|-----|-------|-----------|
| [A] PyTorch MPS 串行 | 165.16s | 0.30 | 30 | 0.06× |
| [B] PyTorch MPS batch=8 | 38.77s | 1.29 | 129 | 0.24× |
| [C] vLLM (vllm-metal) | **9.32s** | **5.36** | **493** | **1.00×** |

> vLLM 相对 MPS 串行加速 **17.7×**，相对 MPS batch=8 加速 **4.2×**。

#### 跨平台对比

| 指标 | NVIDIA RTX 4060 (vLLM) | Mac M4 (vllm-metal) |
|------|------------------------|---------------------|
| 总耗时 | 1.15s | 9.32s |
| QPS | 43.57 | 5.36 |
| tok/s | 3043 | 493 |

> Mac vllm-metal 达到 NVIDIA 的 **1/8 性能**。差距来源：独立显存带宽（272 vs ~120 GB/s）+ CUDA kernel 成熟度。

### 3.3 demo_guided_choice — 枚举约束（12 条）

## 四、综合分析

### 4.1 约束解码效果

```
NVIDIA: stock 60% → 68% → 100%  /  order 42% → 42% → 100%  ← FSM 完美生效
Mac:    stock 90% → 92% → 100%  /  order 52% → 52% → 100%  ← FSM 完美生效（修复后）
```

- `response_format` 仅保证 JSON 语法合法，对字段语义无帮助（order 52%→52%）
- `guided_json` FSM 从 token 级别屏蔽非法输出，实现 **100% schema 合规**
- Mac 裸 prompt 通过率高于 NVIDIA（90%/52% vs 60%/42%），因 vLLM 0.23.0 采样策略差异
- ⚠️ vLLM 0.23.0 将 `guided_json` 参数改为 `structured_outputs`，需适配

### 4.2 吞吐分析

```
MPS 串行  →  MPS batch=8  →  vllm-metal
0.30 QPS  →  1.29 QPS     →  5.36 QPS
           (4.3×)         →  (4.2×)
```

- batch=8 加速 4.3×：消除单条推理的 GPU 空闲
- vllm-metal 再加速 4.2×：PagedAttention + continuous batching 使 GPU 利用率接近满载
- 总加速 17.7× vs 串行

### 4.3 Mac vs NVIDIA 总结

| 维度 | NVIDIA | Mac vllm-metal | 结论 |
|------|--------|---------------|------|
| guided_json 100% | ✅ | ✅ | **一致** |
| PagedAttention | ✅ | ✅ | **一致** |
| Continuous Batching | ✅ | ✅ | **一致** |
| 吞吐 (50 req) | 1.15s | 9.32s | Mac 为 1/8 |
| 安装复杂度 | pip + CUDA 驱动 | 一键 curl 脚本 | Mac 更简单 |

---

## 五、结论

1. **vllm-metal 在 Mac M4 上完整实现了 vLLM 三大核心能力**：PagedAttention + Continuous Batching + xgrammar FSM 约束解码
2. **guided_json FSM 达到 100% schema 通过率**，与 NVIDIA CUDA 版一致
3. **适配要点**：vLLM 0.23.0 将 `extra_body={'guided_json': schema}` 改为 `extra_body={'structured_outputs': {'json': schema}}`，`demo_function_call.py` 已修复
4. **vllm-metal 吞吐达到 NVIDIA 的 1/8**（9.32s vs 1.15s），适合本地开发和学习
5. **全部 6 个原版 demo 脚本基本兼容**，仅需适配 `structured_outputs` 参数名
