# 序列标注（NER）项目

## 1. 项目概述

本项目在 peoples_daily 数据集上对比多种中文命名实体识别方案：

| 路线 | 目录 | 模型 | 变体 |
|------|------|------|------|
| 判别式序列标注 | `src/` | `bert-base-chinese` | BERT + Linear / BERT + CRF |
| 本地生成式微调 | `src_llm/` | `Qwen2.5-0.5B-Instruct` | LoRA SFT（成功）/ 全量微调（失败） |
| API 生成式 | `src_llm/llm_ner.py` | DeepSeek API（`deepseek-v4-pro`） | zero-shot / few-shot |

**数据与评估规模**：

- 训练：train 20,864 条；验证：validation 2,318 条；测试：test 4,636 条。
- BERT 系列在 **validation 全量 2,318 条** 上评估。
- LLM API 与 LoRA SFT 在 **validation 随机采样 100 条** 上评估（span 级 F1，`text.find()` 对齐），便于与生成式路线公平对比。


## 2. 数据分析

### 2.1 数据格式与标签体系

- 每条样本：`tokens`（字符列表）+ `ner_tags`（BIO 标签列表），字符级标注。
- **7 个标签**：`O`, `B-PER`, `I-PER`, `B-ORG`, `I-ORG`, `B-LOC`, `I-LOC`。
- **3 类实体**：人名（PER）、机构（ORG）、地名（LOC）。

### 2.2 分布统计（由 `src/explore_data_excise.py` 与数据文件统计）

| 指标 | 训练集 | 验证集 | 测试集 |
|------|--------|--------|--------|
| 样本数 | 20,864 | 2,318 | 4,636 |
| 实体总数 | 33,992 | 3,819 | 7,707 |
| 句长均值（字） | 46.9 | 47.4 | 47.3 |
| 句长中位数 | 40 | 41 | 41 |
| 句长 P95 | 97 | 97 | 97 |
| 句长最大值 | 574 | 568 | 577 |
| 句长 >128 占比 | 1.65% | 1.64% | 1.53% |
| 每句实体数均值 | 1.63 | 1.65 | 1.66 |
| 每句实体数中位数 | 1 | 1 | 1 |
| 无实体句子占比 | 39.1% | 39.6% | 39.5% |
| 实体长度均值（字） | 3.24 | 3.16 | 3.25 |
| 实体长度中位数 | 3 | 3 | 3 |
| 实体长度 P95 | 8 | 7 | 8 |

**实体类型分布（训练集）**：

| 类型 | 数量 | 占比 |
|------|------|------|
| LOC（地名） | 16,571 | 48.7% |
| ORG（机构） | 9,277 | 27.3% |
| PER（人名） | 8,144 | 24.0% |

验证集上 LOC 略升至 51.1%，PER/ORG 约各占四分之一，**类别不均衡以 LOC 为主**。

**实体长度**：约半数实体为 2–4 字（中位数 3，P95≤8），最长 30 字（训练集）；典型如「中」「华」「美」等单字地名与人名简称。

**每句实体数**：中位数为 1，但存在极端多实体句（训练集单句最多 125 个实体，多为天气稿连排城市名）；约 **四成句子无实体**，训练时负样本（全 `O`）比例高。

## 3. 训练过程

### 3.1 BERT + Linear（`src/train.py`）

- **数据**：`peoples_daily`，7 类 BIO 标签。
- **超参**：`epochs=5`，`batch_size=32`，`max_length=128`，`lr=2e-5`，分类头 `head_lr_mult=5`，`warmup_ratio=0.1`。
- **训练曲线**（`outputs/logs/train_linear.json`）：

| Epoch | train_loss | val_loss | val_entity_f1 | 耗时(s) |
|-------|------------|----------|---------------|---------|
| 1 | 0.163 | 0.020 | 0.929 | 793 |
| 2 | 0.018 | 0.016 | 0.947 | 825 |
| 3 | 0.007 | 0.020 | 0.954 | 833 |
| 4 | 0.004 | 0.021 | 0.957 | 835 |
| 5 | 0.002 | 0.022 | **0.959** | 834 |

**曲线解读**：train_loss 持续下降至 0.002，val_entity_f1 单调升至 0.959；val_loss 在第 3 轮后由 0.016 回升至 0.022，存在**轻度过拟合**迹象，但 F1 仍略升。最优 checkpoint：`outputs/checkpoints/best_linear.pt`。

### 3.2 BERT + CRF（`src/train.py --use_crf`）

- 超参与 Linear 相同；每轮约慢 15%（~900s/epoch）。
- **训练曲线**（`outputs/logs/train_crf.json`）：

| Epoch | train_loss | val_loss | val_entity_f1 | 耗时(s) |
|-------|------------|----------|---------------|---------|
| 1 | 10.057 | 1.038 | 0.933 | 913 |
| 2 | 0.897 | 0.872 | 0.952 | 907 |
| 3 | 0.474 | 0.877 | 0.954 | 907 |
| 4 | 0.251 | 1.186 | 0.962 | 922 |
| 5 | 0.137 | 1.338 | **0.962** | 906 |

**曲线解读**：首轮 CRF 负对数似然尺度大属正常；第 4–5 轮 **val_loss 从 0.87 跳升至 1.34**，而 val_f1 仍升至 0.962，说明 loss 与 span-F1 尺度不一致，但 train_loss 继续下降、val_loss 陡升，**过拟合比 Linear 更明显**。最优 F1 仍高于 Linear 约 0.33 个百分点。Checkpoint：`outputs/checkpoints/best_crf.pt`。

### 3.3 Qwen2.5 LoRA SFT（`src_llm/train_sft.py`）

- **数据**：`peoples_daily` 全量训练集，指令格式 JSON：`{"entities": [{"text": "...", "type": "PER|ORG|LOC"}]}`。
- **模式**：LoRA（约 0.22% 可训练参数），`max_length=256`，`batch_size=4`，`grad_accum=4`。
- **训练曲线**（`outputs/logs/train_sft.json`）：

| Epoch | train_loss | val_loss | 耗时(s) |
|-------|------------|----------|---------|
| 1 | 0.111 | 0.094 | 3238 |
| 2 | 0.072 | 0.087 | 2450 |
| 3 | 0.057 | 0.089 | 3780 |

第 3 轮 val_loss 由 0.087 升至 0.089，**建议早停于第 2 轮**。Adapter：`outputs/sft_adapter/`。

### 3.4 Qwen2.5 全量微调（失败）

本机 **Apple MPS 显存不足**（`MPS backend out of memory`，已分配约 15.36 GiB）。`train_sft.py --full_ft` 三轮 `train_loss` / `val_loss` 均为 **NaN**（`outputs/logs/train_full_ft.json`），无可用 checkpoint。

### 3.5 DeepSeek API NER（`src_llm/llm_ner.py`）

- 模型：`deepseek-v4-pro`；`peoples_daily` validation **随机 100 条**。
- zero-shot：任务说明 + JSON 格式；few-shot：prompt 内 3 条标注示例。
- 结果：`outputs/logs/eval_llm.json`。


## 4. 评估结果与对比

### 4.1 汇总表

**BERT：validation 全量 2,318 条**（`eval_linear_validation.json`、`eval_crf_validation.json`）

| 方案 | Precision | Recall | F1 | 非法序列 |
|------|-----------|--------|-----|----------|
| BERT + Linear | 0.9577 | 0.9606 | **0.9592** | 34 |
| BERT + CRF | 0.9610 | 0.9639 | **0.9625** | 26 |

**生成式：validation 采样 100 条**

| 方案 | Precision | Recall | F1 | 备注 |
|------|-----------|--------|-----|------|
| LLM zero-shot (deepseek-v4-pro) | 0.8476 | 0.4958 | 0.6257 | `eval_llm.json` |
| LLM few-shot (deepseek-v4-pro) | 0.8726 | 0.5153 | 0.6480 | `eval_llm.json` |
| Qwen2.5 LoRA SFT（3 epoch） | 0.8037 | 0.6056 | **0.6908** | `parse_fail=0` |

### 4.2 结果解读

#### （1）BERT：判别式序列标注主导

- CRF 较 Linear：**F1 +0.33%**（0.9625 vs 0.9592），P/R 同步略升。
- **非法 BIO**：Linear 34 条、CRF 26 条（约 1.5% vs 1.1% / 2318），`illegal_start=0`，均为 `I-X` 类型不连续等转移错误；CRF 在子词级 Viterbi 施加转移约束后仍有个别字符对齐还原导致的非法序列。
- 训练集分布（短实体、多 LOC、约 40% 无实体句）与 BERT 预训练中文表征高度匹配，**5 epoch 即达 ~0.96 F1**。

#### （2）CRF vs Linear

| 维度 | Linear | CRF |
|------|--------|-----|
| val F1（第 5 轮） | 0.959 | 0.962 |
| val_loss 趋势（后两轮） | 0.020→0.022（缓升） | 0.877→1.338（陡升） |
| 非法序列 | 34 | 26 |

CRF 通过**全局归一化 + Viterbi** 优化整条标签路径，减少独立 softmax 的局部不一致；收益在本数据集上为**小幅但稳定**（F1 与非法序列双改善）。

#### （3）生成式：高精度、低召回

- API zero-shot：P 0.85，R 0.50，F1 0.63；few-shot：P 0.87，R 0.52，F1 0.65。
- **Precision–Recall 权衡**：模型倾向只输出“有把握”的实体，`pred_total`（210–212）远小于 `gold_total`（359），漏标是 F1 瓶颈。
- few-shot 略优于 zero-shot（+2.2% F1），说明示例有助于类型与格式，**无法弥补召回**。

#### （4）SFT vs API few-shot（同 100 条规模）

- SFT F1 **0.6908** > few-shot **0.6480**；Recall 0.606 > 0.515。
- SFT 在 20,864 条人民日报上监督，**任务对齐**优于 API 仅 3 条示例；`parse_fail=0` 表明 JSON 模板已学稳。
- 仍远低于 BERT 全量 validation（~0.96）：生成式未做 token 级对齐，span 边界与 gold 粒度不一致问题突出。

#### （5）全量微调失败

### 4.3 非法 BIO 序列说明

`src/evaluate.py` 统计：以 `I-X` 开头，或 `B-X`/`I-X` 后接 `I-Y`（X≠Y）的句子数。

- Linear：**34** / 2318（1.47%）
- CRF：**26** / 2318（1.12%），`illegal_start=0`

CRF 在解码层禁止非法转移，但 BERT 子词对齐与截断还原到字符级后，评估口径下仍非零非法。


## 6. 思考题

### 1. SFT 本地小模型 vs LLM API few-shot，谁的 F1 更高？为什么？

**本项目中 SFT 更高。**

- LoRA SFT（100 条）：F1 = **0.6908**，R = 0.6056（`eval_sft.json`）
- DeepSeek few-shot（100 条）：F1 = **0.6480**，R = 0.5153（`eval_llm.json`）

**原因（均基于 peoples_daily 实验）：**

1. **监督规模**：SFT 在 20,864 条新闻语料上学习 PER/ORG/LOC 的 JSON 输出；API 仅 3 条 few-shot，对「新华社+多地名」「单字 LOC」等文体覆盖不足。
2. **召回**：同规模 100 条评估下，SFT Recall **0.6056** 高于 few-shot **0.5153**；API 虽 Precision 更高（0.87 vs 0.80），但漏标更多，F1 反而更低。
3. **格式**：SFT `parse_fail=0`，错误主要在 span 对齐而非 JSON 崩溃。

### 2. NER 的 JSON 输出比分类的单词输出难控制，体现在哪里（parse_fail 数）？

分类输出单个标签，解码空间小。NER 需合法 JSON、多实体列表、字段名与类型枚举，输出更长、语法约束更多。

**本项目（peoples_daily + `evaluate_sft.py`）**：

- LoRA SFT 评估 100 条：**`parse_fail = 0`**，说明模板已固化。
- API 路径无单独 `parse_fail` 计数；`json.loads` 失败时等价空实体（该条 R=0）。从 `eval_llm.json` detail 看，多数回复格式正确，错误以**漏标、span 过长、LOC/ORG 混淆**为主。

结论：SFT 将**格式风险**压到 0，但 F1 仍受 **Recall 与边界粒度** 主导（0.69 vs BERT 0.96）。

### 3. BERT+CRF 保证零非法序列，生成式 NER 有这个保证吗？如何处理？

**理论**：CRF 转移矩阵禁止 `O→I-LOC`、`B-PER→I-ORG` 等，Viterbi 解码的子词级路径满足 BIO 语法。

**本项目实测（validation 2318 条）**：CRF 仍有 **26** 条非法转移（Linear 34），与 BERT 子词对齐、截断、字符级还原有关，**非评估口径下的零非法**。

**生成式 JSON**：无 BIO 合法性；可能出现原文不存在的实体、类型错误、粒度与 gold 不一致。

**处理手段（本项目已用或可扩展）**：

1. Prompt + few-shot 固定 schema；
2. SFT 监督目标 JSON（`parse_fail→0`）；
3. 后处理：`json.loads` + `text.find()` 对齐原文；
4. 可选：类型白名单、去重、过滤无法匹配的 span；若要接近 CRF 确定性，需约束解码或改回序列标注头。

### 4. 若给 SFT 模型也在系统提示中加入 few-shot 示例，F1 会提升吗？

**可能小幅提升，上限仍远低于 BERT+CRF。**

**可能收益**：强化「单字 LOC」「嵌套机构」等 peoples_daily 标注习惯；减少对空 `entities` 的过度使用。

**收益有限的原因**：

1. SFT 已在 **20,864 条** 上学习，prompt 内 3 条示例的信息量远小于训练集；API zero→few 仅 +2.2% F1（0.626→0.648），SFT 边际可能更小。
2. 主误差是 **Recall 与 span 粒度**（见 §4.3），需更多 epoch、更大模型或判别式蒸馏，而非仅 prompt。
3. 更长 prompt 占用 `max_length=256`，对 P95 句长 97 的多实体稿不利。


## 7. 项目总结：

### 7.1 序列标注理论在本项目中的体现

- **BIO 标注**：将 NER 化为逐 token 分类；`B-` 起、`I-` 续、`O` 非实体，便于与 CRF、softmax 头结合。
- **CRF 全局归一化**：在 Linear 发射分数上，用转移矩阵对整条序列打分，训练最大化真实路径似然；推理 **Viterbi** 求最优路径，显式惩罚非法 `B-I` 转移。
- **与实验的对应**：CRF F1 高于 Linear、非法序列 26 vs 34，体现了**结构化解码**相对独立分类的优势；残余非法来自 BERT 子词与字符 gold 的对齐缝隙。

### 7.2 判别式 vs 生成式 NER

| | 判别式（BERT+CRF） | 生成式（API / SFT） |
|--|-------------------|---------------------|
| 输出 | 每 token 一个 BIO 标签 | 自由文本 JSON |
| 监督 | token 级对齐，2 万+ 条 | span 列表；SFT 全量，API 无梯度 |
| 约束 | CRF 转移 + 词表 | 仅 prompt/SFT 模板 |
| 本项目 F1 | ~0.96（2318 条） | 0.63–0.69（100 条） |
| 主要误差 | 少量非法 BIO、边界 | 漏标、粒度过粗/过细 |

### 7.3 LoRA SFT vs API prompting

- **SFT**：离线、可部署、`parse_fail=0`、F1 0.69；需 GPU 时间与适配器存储。
- **API few-shot**：零训练、F1 0.65、P 更高但 R 更低；依赖网络、成本与延迟。
- **权衡**：要接近 BERT 精度需更大模型 + 更长 SFT 或 **蒸馏 BERT→生成式**；仅 prompt 无法弥补 2 万条监督的信息差。
