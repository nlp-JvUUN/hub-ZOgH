## 1. 项目背景与目标

在大模型应用开发中，让模型"动手"调用外部工具是实现落地价值的关键能力。当前主流的工具调用方式有三种：**Function Call**（模型原生能力）、**MCP**（协议标准）、**CLI**（命令行工具）。

本项目以 **AI技术面试智能问答系统** 为业务场景，实现同一份业务后端 × 三种工具调用方式的完整对比实践，量化评估各方案在**接入成本、性能延迟、安全性、跨模型复用性**四个维度的差异，为企业级选型提供可落地的参考数据。

**核心业务能力**：
- **RAG检索**：基于FAISS向量数据库检索7份经典AI技术论文（Transformer/BERT/GPT-3/InstructGPT/LLaMA/RAG/动手学深度学习）
- **天气查询**：基于Open-Meteo API实现城市天气实时查询与预报

**项目价值**：
- 提供可复现的工具调用对比基准，避免选型时的主观臆断
- 沉淀工程实践经验，包括多轮工具调用循环、沙箱安全设计、跨进程通信等
- 作为大模型开发工程师求职的项目成果展示

## 6. 各方案原理简介

### 6.1 Function Call（模型原生函数调用）

**核心原理**：开发者手写工具的JSON Schema（name/description/parameters），模型根据用户问题自主决策是否调用工具及传递参数。

**流程**：
1. 构建工具Schema列表
2. 将Schema传入模型`chat.completions.create(tools=)`
3. 模型输出`tool_calls`（或直接回答）
4. 宿主解析tool_calls，调用对应后端函数
5. 将工具结果以`role=tool`回填到对话历史
6. 循环直到模型不再调用工具，输出最终回答

**优点**：接入成本最低，进程内直调性能最优
**缺点**：Schema需手写，跨模型兼容性差

### 6.2 MCP（模型上下文协议）

**核心原理**：通过标准协议实现工具的自动发现和跨进程调用，工具提供者只需实现Server，模型侧只需实现Client。

**流程**：
1. Host启动多个MCP Server子进程（stdio通信）
2. 每个Server完成`initialize()`握手 + `list_tools()`工具发现
3. Host将MCP的`inputSchema`转换为OpenAI格式的`parameters`
4. 模型输出tool_calls
5. Host通过`ClientSession.call_tool()`跨进程调用工具
6. 结果回填，循环直到最终回答

**优点**：工具自动发现，跨产品复用，写一次到处接
**缺点**：需实现Server，多一层IPC开销

### 6.3 CLI（命令行工具）

**核心原理**：将工具能力封装为命令行程序，模型通过生成命令字符串来调用。

**两种形态**：
| 形态 | 原理 | 安全性 |
|------|------|--------|
| **具名run_cli** | 模型调用`run_cli(command, args)`，command是白名单enum | 高（白名单限制） |
| **通用run_bash** | 模型自己拼shell命令，沙箱检查后执行 | 中（依赖沙箱） |

**沙箱设计**：
- 危险命令黑名单（rm/del/format/sudo/curl\|sh/nc等）
- 命令头白名单（fincli/python/git/ls/cat/echo）
- 15s执行超时
- 工作目录锁定

**优点**：与模型完全无关，复用现有命令生态
**缺点**：subprocess开销最大，bash形态需沙箱兜底

## 9. 结果分析与讨论

### 9.1 工具调用一致性

四种方式对同一问题调用的工具与参数基本一致，说明底层业务能力相同，差异仅在于接入方式。

### 9.2 性能差异分析

- **Function Call最优**：进程内直调后端函数，无额外开销
- **MCP次之**：多一层stdio IPC和Server启动开销，但工具发现能力最强
- **CLI(named)**：subprocess启动开销，但白名单机制安全可控
- **CLI(bash)**：开销最大，模型需多次尝试才能生成有效命令

### 9.3 幻觉控制行为

所有方式都能正确拒绝不在知识库的问题（区块链），但工具调用策略有所不同：
- Function Call/CLI(named)：0工具调用直接拒绝
- MCP/CLI(bash)：先调用`list_papers`确认库内内容再拒绝

### 9.4 多轮调用必要性

对比实验发现，部分复杂问题（如BERT vs GPT-3对比）需要模型多次检索才能获取充分信息。单轮闭环无法满足此需求，必须实现多轮工具调用循环（本项目已实现）。

## 10. 最终结论

### 10.1 选型建议

| 场景 | 推荐方案 | 理由 |
|------|---------|------|
| 快速原型开发 | Function Call | 接入成本最低，性能最优 |
| 多工具生态构建 | MCP | 工具自动发现，跨产品复用 |
| 工程师日常工具 | CLI(named) | 白名单安全，命令行友好 |
| 通用工具调用 | CLI(bash) | 零封装，但需完善沙箱 |

### 10.2 核心发现

1. **多轮工具调用是必须的**：复杂问题需要模型多次交互才能给出完整回答
2. **沙箱设计不可少**：CLI(bash)形态必须有完善的安全兜底机制
3. **协议层解耦价值大**：MCP实现了工具与模型的解耦，提升复用性
4. **业务逻辑应分离**：共享后端确保对比公平性，也是企业级开发的最佳实践

## 12. 常见问题

### Q1：为什么MCP方式启动时报`ModuleNotFoundError`？

**原因**：虚拟环境依赖冲突。**解决**：清除`PYTHONPATH`环境变量后运行。

```bash
$env:PYTHONPATH=""
python mode_mcp/run_mcp.py -q "问题"
```

### Q2：CLI(bash)提示找不到`fincli`命令？

**原因**：虚拟环境Scripts目录不在PATH中。**解决**：确保已执行`pip install -e .`，或检查环境变量配置。

### Q3：天气查询返回错误？

**原因**：Open-Meteo API响应格式变更。**解决**：项目已修复坐标字段路径问题，确保使用最新代码。

### Q4：FAISS索引读取失败？

**原因**：Windows中文路径处理问题。**解决**：项目已改为相对路径读取，确保在项目根目录执行。

### Q5：模型不调用工具直接回答？

**原因**：系统提示词约束不足。**解决**：提示词已添加"必须基于检索内容回答"的强制约束。

### 13.1 架构设计建议

```
┌─────────────────────────────────────────────────────────────┐
│                    用户请求入口                              │
│            API Gateway / CLI / Web Interface                │
└──────────────────────┬──────────────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────────────┐
│                    LLM调度层                                │
│        Function Call / MCP / CLI 三种方式并行支持            │
└──────────────────────┬──────────────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────────────┐

│         工具注册中心 / 权限控制 / 调用审计 / 缓存            │
└──────────────────────┬──────────────────────────────────────┘
                       ▼
┌─────────────────────────────────────────────────────────────┐
│                    业务工具层                                │
│         RAG检索 / 天气查询 / 数据库查询 / API调用            │
└─────────────────────────────────────────────────────────────┘
```

### 13.2 安全最佳实践

1. **最小权限原则**：工具权限按需分配，避免过度授权
2. **调用审计**：记录所有工具调用日志，便于追溯
3. **输入校验**：对工具参数进行严格校验，防止注入攻击
4. **超时控制**：设置工具执行超时，防止资源耗尽
5. **隔离执行**：CLI工具在沙箱中执行，限制文件系统访问

### 13.3 性能优化建议

1. **索引预热**：FAISS索引在服务启动时预加载
2. **工具缓存**：对相同参数的工具调用结果进行缓存
3. **并行执行**：支持多个工具并行调用，减少总耗时
4. **连接池复用**：HTTP连接池、MCP会话复用
5. **异步化**：后端函数支持异步调用，提升吞吐量

### 13.4 监控与告警

| 监控指标 | 告警阈值建议 |
